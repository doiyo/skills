# X Post Style

Use these defaults unless the user or repo supplies stronger guidance.

## Tone

- Write plain builder updates: specific, useful, and lightly promotional.
- Lead with the user-facing change or practice/workflow benefit.
- Explain why it matters in one or two short lines.
- Use the product name and CTA URL only after inferring them from repo/user context.
- Avoid hype, vague launch language, and implementation jargon.

## Format

- Prefer standalone posts over threads unless the user asks for a thread.
- Keep each post under X's character limit with room for link shortening and edits.
- Use short paragraphs or compact bullets when they improve scanning.
- Include a light CTA when natural, such as a product URL or "try it now" phrasing adapted to the product.
- Avoid hashtags by default. Add at most 1-3 only when the account already uses them or the user asks.

## Topic Selection

Good topics are user-facing and concrete:

- New settings or controls users can act on.
- Clear visual or workflow improvements.
- Review, navigation, timeline, or performance changes users can feel.
- Help/docs/site updates that clarify a core workflow.

Avoid standalone posts for:

- Pure dependency updates.
- Deployment plumbing.
- Test-only changes.
- Refactors with no user-visible effect.

## Media

- Attach one relevant screenshot per post when possible.
- Crop to show the feature, not the whole app shell, unless context is needed.
- Prefer square or 4:5 crops for feed readability, but keep the UI legible over strict aspect ratio.
- Use existing screenshots before capturing new ones.
